## Why

The integer-to-UUID migration (20250718) generated photo IDs in the format `00000000-0000-0000-0000-000000XXXXXX` by zero-padding old integer IDs. These are valid PostgreSQL UUIDs but fail the `uuid` library's `validate()` function, which enforces RFC 4122 version/variant bits. This causes runtime errors (`Wrong UUID format for photoId`) when users interact with migrated photos — currently breaking `removePhotoFromWave` and `addPhotoToWave`.

Only `photoId` values are affected — all other UUIDs (`uuid`, `waveUuid`, `friendshipUuid`, `chatUuid`, etc.) are generated by `uuidv4()` and are valid RFC 4122. Those should continue using strict validation.

## What Changes

- Create a shared `isValidPhotoId` utility in `lambda-fns/utilities/` that checks UUID shape via regex without enforcing RFC 4122 version/variant bits
- Replace `uuidValidate` with `isValidPhotoId` for `photoId` parameter validation in the 2 broken controllers (`addPhoto.ts`, `removePhoto.ts`)
- Add `isValidPhotoId` validation for `photoId` to the 9 other controllers that receive it from GraphQL but currently have no validation, for consistency and input safety
- All other UUID validations remain unchanged (strict RFC 4122 via `uuid.validate`)

## Capabilities

### New Capabilities
- `photo-id-validation`: Relaxed UUID format validation specifically for photo IDs that may have been migrated from integer values

## Impact

- **Lambda code**: 11 controller files updated — 2 replace existing `uuidValidate` for photoId, 9 add new `isValidPhotoId` validation
  - `waves/addPhoto.ts`, `waves/removePhoto.ts` (fix broken validation)
  - `photos/delete.ts`, `photos/watch.ts`, `photos/unwatch.ts`, `photos/getPhotoDetails.ts`, `photos/getPhotoAllCurr.ts`, `photos/getPhotoAllNext.ts`, `photos/getPhotoAllPrev.ts` (add validation)
  - `abuseReports/create.ts`, `comments/create.ts` (add validation)
- **New file**: `lambda-fns/utilities/isValidPhotoId.ts`
- **No API changes**: Error messages and behavior remain the same, validation is just more permissive for `photoId`
- **No database changes**: No migrations needed
